Incorporating Libraries Into Your Code

Squinter lets you incorporate multiple files into your project. Electric Imp applications — called ‘Models’ — comprise two files: one for device code, the other for cloud-based agent code, but Squinter lets you embed into them links to other files, such as libraries.

For example, you might want to use the same library in multiple models — they all use the same sensor, say. Sharing a single library file allows you to update that file without having to edit every model that uses it. You just recompile each project to incorporate the changes you’ve made.

Squinter expects your model files to be named in the standard Electric Imp schema: *.agent.nut and *.device.nut for, respectively, your agent and device code. The two identifiers (represented here by the wildcard *) need not be identical, but when creating a new project from source code files, Squinter will use the agent code filename as the basis for the project’s name.

Local library files should be entered into your agent and device source code using the following syntax:

#import "library_filepath\library_filename"

This is to distinguish these libraries from Electric Imp online libraries, which use the #require directive.

The name of the library file is arbitrary, but *.class.nut and *.library.nut are the recommended forms. You may include a full Unix filepath; if you only provide a filename, Squinter expects the file to reside in the same folder as the project file (see ‘Working with Projects’)
